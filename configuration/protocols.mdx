Traccar listens for incoming connections from GPS tracking devices on protocol-specific TCP or UDP ports. Each protocol decoder is independent; enabling a protocol is as simple as adding a single `<entry>` to `traccar.xml` with the port number. Traccar ships with decoders for over 267 protocols, covering thousands of device models from hundreds of manufacturers worldwide.

<Note>
  After adding or changing protocol ports in `traccar.xml`, you must restart the Traccar server and open the new port(s) in your server's firewall.
</Note>

---

## How protocol configuration works

Every protocol has a lowercase name prefix (e.g. `teltonika`, `gt06`, `osmand`). Append a configuration suffix to that prefix to set a parameter for that protocol.

### Configuration suffixes

| Suffix | Type | Description |
|--------|------|-------------|
| `<protocol>.port` | integer | **Required to enable the protocol.** TCP (or UDP) port number to listen on. |
| `<protocol>.address` | string | Network interface to bind to. If not specified, the server binds to all interfaces. Multiple addresses (IPv4 and IPv6) can be supplied as a comma-separated list. |
| `<protocol>.ssl` | boolean | Enable SSL/TLS on this protocol listener. Not all protocols support SSL. |
| `<protocol>.timeout` | integer | Per-protocol connection timeout in seconds. Overrides `server.timeout` for this protocol. |
| `<protocol>.devicePassword` | string | Device-level password used for authenticating commands sent to the device. |
| `<protocol>.devices` | string | Comma-separated list of unique device IDs. Used only for polling protocols where the server initiates contact. |
| `<protocol>.interval` | long | Polling interval in seconds. Used only for protocols that require the server to poll devices. |
| `<protocol>.extended` | boolean | Enable extended functionality. Some devices support richer data formats that are disabled by default for compatibility. |
| `<protocol>.utf8` | boolean | Decode string fields as UTF-8 instead of ASCII. Only applicable for some protocols. |
| `<protocol>.can` | boolean | Enable CAN bus data decoding. Not supported by all devices. |

## Example configuration

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE properties SYSTEM 'http://java.sun.com/dtd/properties.dtd'>
<properties>

    <!-- Database (required) -->
    <entry key='database.driver'>org.h2.Driver</entry>
    <entry key='database.url'>jdbc:h2:./data/database</entry>
    <entry key='database.user'>sa</entry>
    <entry key='database.password'></entry>

    <!-- OsmAnd / GPS Logger HTTP protocol -->
    <entry key='osmand.port'>5055</entry>

    <!-- Teltonika FMB/FMT/FMP/FMC series (TCP + UDP) -->
    <entry key='teltonika.port'>5027</entry>

    <!-- GT06 / GT02 family (Concox, Jimi IoT, iStartek, Gosafe) -->
    <entry key='gt06.port'>5023</entry>

    <!-- Queclink GL/GV/GS series -->
    <entry key='gl200.port'>5004</entry>

    <!-- Ruptela FM-Eco / FM-Pro / FM-Tco -->
    <entry key='ruptela.port'>7354</entry>

    <!-- Coban TK-103 compatible devices -->
    <entry key='tk103.port'>5002</entry>

    <!-- Galileosky v4/v5/v6/v7 -->
    <entry key='galileo.port'>7001</entry>

    <!-- Wialon IPS -->
    <entry key='wialon.port'>20332</entry>

    <!-- Meitrack T1/T3/MVT series -->
    <entry key='meitrack.port'>5020</entry>

    <!-- Enfora (Sierra Wireless) -->
    <entry key='enfora.port'>5016</entry>

</properties>
```

<Tip>
  To verify that a device is successfully connecting and sending data, open the Traccar web interface, navigate to **Settings → Devices**, select your device, and watch the **Last Update** timestamp. You can also monitor the server log at `./logs/tracker-server.log`.
</Tip>

## UDP protocols

Some protocols operate over UDP instead of TCP (e.g. many OBD and personal tracker protocols). The configuration key is identical — Traccar automatically selects the correct transport layer based on the protocol's implementation. No extra configuration is needed:

```xml
<!-- UDP protocol — configuration is the same as TCP -->
<entry key='osmand.port'>5055</entry>
```

Some protocols support both TCP and UDP simultaneously on the same port.

---

## Polling protocols

A small number of protocols are "polling" protocols where the server must initiate communication with the device (rather than the device calling in). Configure these with the `.devices` and `.interval` suffixes:

```xml
<!-- Poll devices every 60 seconds -->
<entry key='myprotocol.port'>5099</entry>
<entry key='myprotocol.devices'>device1_id,device2_id,device3_id</entry>
<entry key='myprotocol.interval'>60</entry>
```

The `.devices` value is a comma-separated list of unique device identifiers as they appear in the Traccar database. The `.interval` value is in seconds.

## Firewall configuration

After enabling a protocol, open the corresponding port in your OS firewall:

<Tabs>
  <Tab title="firewalld (RHEL/CentOS)">
```bash
sudo firewall-cmd --permanent --add-port=5027/tcp
sudo firewall-cmd --permanent --add-port=5055/tcp
sudo firewall-cmd --reload
```
  </Tab>
  <Tab title="ufw (Ubuntu/Debian)">
```bash
sudo ufw allow 5027/tcp
sudo ufw allow 5055/tcp
sudo ufw reload
```
  </Tab>
  <Tab title="iptables">
```bash
sudo iptables -A INPUT -p tcp --dport 5027 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 5055 -j ACCEPT
sudo iptables-save > /etc/iptables/rules.v4
```
  </Tab>
</Tabs>

<Note>
  If Traccar is behind a NAT gateway or cloud load balancer, make sure both the OS firewall and the network security group/ACL allow the device ports. Devices must be able to reach the public IP of your Traccar server on the configured port.
</Note>
