Traccar processes incoming GPS data from devices and stores each fix as a `Position` record. Positions contain coordinates, speed, course, altitude, and a rich set of telemetry attributes parsed from the device protocol. Every position is associated with a device and carries three timestamps that let you distinguish when the fix was taken, when the device reported it, and when the server received it.

## Position Fields

The core fields of the `Position` model (`Position.java`) are:

| Field | Type | Description |
|---|---|---|
| `id` | long | Auto-assigned unique identifier |
| `deviceId` | long | ID of the reporting device |
| `protocol` | string | Name of the protocol used to receive the data |
| `serverTime` | date | When the Traccar server received the message |
| `deviceTime` | date | Timestamp reported by the device firmware |
| `fixTime` | date | When the GPS fix was acquired |
| `valid` | boolean | Whether the fix is considered accurate by the device |
| `latitude` | double | Latitude in decimal degrees (−90 to 90) |
| `longitude` | double | Longitude in decimal degrees (−180 to 180) |
| `altitude` | double | Altitude in meters above sea level |
| `speed` | double | Speed in **knots** as received from the device |
| `course` | double | Heading in degrees (0–360, clockwise from north) |
| `accuracy` | double | Estimated accuracy radius in meters |
| `address` | string | Reverse-geocoded street address (if enabled) |
| `network` | object | Cell tower and Wi-Fi network data used for geolocation |
| `geofenceIds` | array | IDs of geofences the device is currently inside |

<Note>
Speed stored in the `Position` record is in **knots** as parsed from the device protocol. The web interface converts this to km/h or mph based on user preferences.
</Note>

## Live Position Updates

The Traccar web interface uses a persistent connection (WebSocket or server-sent events depending on the client) to receive new positions in real time. Each time a device reports a new position, connected clients receive a push notification containing the updated `Position` object.

For API-based integrations, poll for the current position with:

```bash
GET /api/positions?deviceId={id}
```

This returns the **latest** known position for the device. To stream historical positions over a time range, add `from` and `to` parameters (see examples below).

## Valid vs. Invalid Fixes

The `valid` boolean reflects whether the device's GPS receiver reported a valid fix at the time of the reading. Devices may send positions with `valid: false` when they have no satellite signal but still want to report I/O state or alarms.

By default, Traccar stores all positions including invalid ones. To suppress invalid position storage, set `database.saveInvalid=false` in `traccar.xml`. To also save empty (no-attribute) positions, set `database.saveEmpty=true`.

## Data Forwarding

Traccar can forward every incoming position to an external HTTP endpoint in real time. Configure the following keys in `traccar.xml`:

```xml
<entry key='forward.url'>https://example.com/webhook</entry>
<entry key='forward.header'>Authorization: Bearer token123</entry>
<entry key='forward.json'>true</entry>
```

When `forward.json` is `true`, the full position JSON is posted to the URL. When `false`, fields are sent as URL-encoded form parameters.
